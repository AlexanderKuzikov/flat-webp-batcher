# flat-webp-batcher

Node.js-утилита для пакетной обработки изображений из вложенных папок с фильтрацией по количеству файлов, уменьшением по ширине и конвертацией в WebP.

## Что делает

Утилита:
- рекурсивно обходит `inputDir`;
- выбирает только те папки, в которых найдено не меньше `minImagesInFolder` изображений;
- обрабатывает изображения параллельно;
- уменьшает ширину до `maxWidth` пикселей без upscale;
- сохраняет пропорции по высоте;
- конвертирует результат в WebP с качеством `webpQuality`;
- складывает все результаты в одну плоскую папку `outputDir`;
- не сохраняет исходную структуру подпапок;
- автоматически разрешает коллизии имен через суффиксы `__2`, `__3` и так далее;
- умеет читать `config.json` в строгом и tolerant-режиме.

## Конфигурация

Файл `config.json`:

```json
{
  "inputDir": "IN",
  "outputDir": "OUT",
  "maxWidth": 1024,
  "webpQuality": 75,
  "minImagesInFolder": 6,
  "concurrency": 0,
  "supportedExtensions": [
    ".jpg",
    ".jpeg",
    ".png",
    ".webp",
    ".tif",
    ".tiff",
    ".bmp",
    ".gif",
    ".avif"
  ]
}
```

### Параметры

- `inputDir` — входная папка; можно указывать относительный или абсолютный путь.
- `outputDir` — выходная папка; можно указывать относительный или абсолютный путь.
- `maxWidth` — максимальная ширина итогового изображения.
- `webpQuality` — качество WebP от 1 до 100.
- `minImagesInFolder` — минимальное число изображений в папке, чтобы она попала в обработку.
- `concurrency` — количество одновременно обрабатываемых файлов; если `0`, утилита выбирает значение автоматически.
- `supportedExtensions` — список расширений, которые считаются изображениями.

## Пути

Поддерживаются:
- относительные пути;
- абсолютные пути;
- пути на другом диске;
- прямые и обратные слэши;
- кириллица, пробелы, точки, дефисы и другие обычные символы в имени папок и файлов.

Примеры для Windows:

```json
{
  "inputDir": "D:/source/images",
  "outputDir": "E:/result/webp"
}
```

или:

```json
{
  "inputDir": "D:\\source\\images",
  "outputDir": "E:\\result\\webp"
}
```

Примеры для Linux/macOS:

```json
{
  "inputDir": "/mnt/data/source",
  "outputDir": "/mnt/data/out"
}
```

## Tolerant-режим для config.json

По умолчанию утилита сначала пытается разобрать `config.json` как обычный строгий JSON.

Если строгий JSON не разбирается, утилита автоматически пробует tolerant-режим. Это полезно для Windows-путей, записанных в конфиге как есть, без двойного экранирования обратных слэшей.

Строго совместимый вариант:

```json
{
  "inputDir": "C:\\-=Букеты=-\\1. Парсинг",
  "outputDir": "D:\\OUT",
  "maxWidth": 1024,
  "webpQuality": 75,
  "minImagesInFolder": 6,
  "concurrency": 0,
  "supportedExtensions": [".jpg", ".jpeg", ".png"]
}
```

Упрощенный Windows-вариант:

```json
{
  "inputDir": "C:\-=Букеты=-\1. Парсинг",
  "outputDir": "D:\OUT",
  "maxWidth": 1024,
  "webpQuality": 75,
  "minImagesInFolder": 6,
  "concurrency": 0,
  "supportedExtensions": [".jpg", ".jpeg", ".png"]
}
```

Если включился tolerant-режим, утилита покажет одно предупреждение в консоли и продолжит работу.

## Логика отбора папок

Обрабатываются только те папки, в которых найдено не меньше `minImagesInFolder` изображений.

Пример:
- `IN/a` содержит 3 изображения — папка пропускается.
- `IN/b` содержит 6 изображений — все изображения из этой папки попадают в обработку.
- `IN/c/nested` содержит 8 изображений — папка `nested` попадает в обработку независимо от соседних папок.

## Коллизии имен

Все результаты сохраняются в одну плоскую папку без структуры подпапок.

Если в разных папках встречаются одинаковые имена файлов, утилита не перезаписывает результат, а добавляет суффикс:

- `photo.jpg` → `photo.webp`
- второй `photo.png` → `photo__2.webp`
- третий `photo.tif` → `photo__3.webp`

## Поведение обработки

- EXIF orientation учитывается.
- Маленькие изображения не увеличиваются.
- Если `outputDir` находится внутри `inputDir`, папка результата исключается из сканирования.
- Если `outputDir` не существует, она создается автоматически.
- Если `inputDir` не существует, утилита завершится с ошибкой.

## Вывод в терминал

CLI не печатает строку на каждый успешно обработанный файл.

Поведение вывода:
- в обычном терминале показывается аккуратный live progress bar;
- отображаются общий прогресс, количество успешных и неуспешных файлов, текущий файл и значение `concurrency`;
- в конце выводится компактная сводка;
- список ошибок печатается только после завершения обработки.

Если stdout не является TTY, утилита не пытается рисовать анимированный прогресс-бар и переходит в спокойный текстовый режим.
